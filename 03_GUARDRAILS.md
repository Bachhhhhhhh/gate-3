# Gate 3 Deliverable: Guardrails

This file documents the safety, product, and AI guardrails for Inspectra.

## AI Output Guardrails

The VLM output is treated as inspection assistance, not final truth.

Rules:

- VLM results are screening evidence only.
- VLM confidence is not a final pass/fail quality standard.
- Reports should be reviewed by a human inspector before final acceptance.
- The UI marks suspicious regions and stores evidence, but a reviewer confirms
  the result.
- False negatives are expected based on evaluation recall, so "no defect" does
  not mean the surface is guaranteed safe.

## Report Review Guardrails

The web app includes a human review flow:

- Reports are associated with the authenticated user through `created_by`.
- Normal users see their own reports.
- Admin users can inspect all reports.
- Report detail pages include a reviewer note field.
- The "Xac nhan ket qua" / confirmation action records reviewer state.
- The reports list shows a confirmation flag so operators can tell which reports
  have been reviewed.

## Data and Access Guardrails

Authentication and authorization:

- Users log in with email/password.
- Backend uses JWT auth.
- Roles: `admin`, `inspector`, `operator`.
- Non-admin users are restricted to their own reports.
- Admin users can manage and review broader report history.

Artifact handling:

- Large artifacts are stored as files.
- Database stores metadata and artifact paths.
- API keys are environment variables and must not be committed.
- Uploaded images, overlays, segmentation images, and result JSON are stored as
  report evidence.

## Robot and Camera Guardrails

Camera images are not used for robot navigation decisions in this MVP.

Navigation boundary:

```text
CAD/floorplan -> deterministic planner -> safety gate -> robot controller
```

Rules:

- Camera frames are used for inspection capture only.
- Camera frames do not decide where the robot drives.
- Qwen3.6-35B-A3B can only provide upper-level advice such as scan order,
  standoff, capture spacing, or planner parameters.
- AI-generated executable waypoints must not be sent directly to the robot.
- Executable waypoints must be generated by deterministic CAD/floorplan logic
  and filtered by a safety gate.

## Demo Guardrails

For Gate 3 demo:

- Robot movement footage should be described as manual-control hardware demo if
  it is not fully autonomous.
- Web upload and ESP32-CAM inspection are valid inspection input sources.
- The demo should not claim solved full autonomous navigation from arbitrary
  drawings.
- The demo should claim constrained CAD-guided / AI-assisted inspection planning
  only.

## Failure Mode Handling

Expected failure modes:

- ESP32-CAM IP changes or camera is on a different network.
- VLM misses fine cracks.
- VLM may classify peeling/spalling inconsistently.
- Novita API latency can vary.
- Uploaded image quality may be too low for reliable inspection.

Mitigations:

- Show image quality and reviewer notes in the report.
- Keep source image and result JSON for audit.
- Use segmentation/overlay as visual evidence, not final proof.
- Keep manual review in the loop.
- Use cached/local outputs when possible during demos to control cost.
